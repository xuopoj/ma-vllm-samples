## 一、现状:用户当前的多机部署流程

先看一下今天用户在平台上做一次多机部署,实际要走的流程。

**以多机混合引擎部署为例:**

1. 配置第一个部署单元:用主节点脚本启动 vllm 引擎(`dp-address` 设为本节点 IP);
2. 配置第二个部署单元:用从节点脚本启动 vllm 引擎(`dp-address` 设为主节点 IP——
   而这个 IP 需要**自己解析 rank table、取出第一个节点的地址**才能拿到);
3. 启动部署。

**以 P-D 分离部署为例:**

1. 配置第一个部署单元:执行 P 节点的启动脚本(若一个 P 实例跨多个节点,还要**基于
   rank table 取出 P 的首节点 IP** 并注入脚本);
2. 配置第二个部署单元:执行 D 节点的启动脚本;
3. 在第一个部署单元里再启动一个 proxy,而 proxy 需要列出**所有 P/D 节点的地址**——
   这些同样得**靠解析 rank table 拿到**。

这套流程暴露出两个问题:

- **太复杂、一线可操作性差。** 用户要在部署态自己感知角色,为每个角色分别指定命令和
  环境信息,认知负担重,容易配错。
- **rank table 要反复手动解析。** 同一份 rank table,在不同角色、不同环节被一次次重新
  解析、各取所需,每次都是手搓——出错概率大,出错后还难定位、难排查。

还有一点:**上述流程只在推理 2.0 上跑得通。** 推理 1.0 整个部署只有一个部署单元,
无法区分单元、给不同角色下发不同的启动命令,因此这套"按角色分别配置"的做法在 1.0 上
根本无法实现。

## 二、问题的本质:运行时集群发现这一步,平台没替用户做

上面流程的复杂,根子在一个**平台留给每个用户自己解决的死结**:**平台把同一条启动命令
下发到每个节点,但节点并不知道"我是谁"**——谁当 leader 起 API server、谁当 headless
worker 去握手?leader 的地址又是运行时才分配的,写脚本时根本不存在,无法硬编码。

这个死结在**推理 1.0 和 2.0 上都存在**:

- 1.0 只能给所有节点下发同一条命令,连分角色都做不到(见上节);
- 2.0 虽支持多角色配置,但 worker 启动时 leader 地址仍是运行时才确定,平台不会替用户注入。

结果就是上一节看到的:**每个用户都得自己去解析
`/user/global/config/global_rank_table.json`**,自己判断本节点 rank、自己提取 leader
地址、自己处理 group 路由约束。这件事没有标准答案,各做各的,极易出错。

> 一个很说明问题的例子:我们看到一份**生产环境的部署手册,里面手写了一份几乎一模一样的
> rank table 解析逻辑**——也就是说,这不是个别现象,用户正在现场各自重复造轮子,而且会
> 踩到各种隐蔽的坑(握手超时、KV 端口冲突、group 0 路由配错等)。

## 三、两点建议

这两点正好对应上面的两个问题。

### 建议一:平台侧解析 rank table,并注入标准环境变量(解决"反复手搓解析")

与其让每个用户各自解析 rank table,**平台应当解析一次,把结果以一组标准环境变量注入到
每个节点的容器里**。我们在实现中沉淀出的一套变量约定(可直接作为规格参考):

| 变量 | 含义 |
|------|------|
| `*_NODE_RANK` | 本节点在扁平节点列表中的序号(0 起) |
| `*_NNODES` | 总节点数 |
| `*_GROUP0_SIZE` | group 0(可路由流量)的节点数 |
| `*_CURRENT_ADDR` | 本节点 IP |
| `*_ADDR_<N>` | 第 N 个 rank 的 IP(worker 据此找 leader) |
| `MASTER` | leader 上为 true,其余 false |

有了这组变量,用户脚本就不必再碰 rank table,直接按 rank 分流即可。

### 建议二:平台侧提供多机部署脚本脚手架(解决"部署态分角色太复杂")

我们做下来发现,从单机到 P-D 分离,**"监听 rank table → 解析 → 按角色分发"这套骨架是
完全一致的,变的只是最后的 `vllm serve` 参数块**。这意味着平台完全可以提供一套部署脚手架:
一个统一的入口脚本(按 rank 自动分流到对应角色),用户只需要填写自己模型的启动参数即可,
不必从零手写多机编排、网卡绑定、socket 配置这些公共逻辑。

这样用户就不必再在部署态逐个角色配置命令和环境,一条命令下发到所有节点即可,
把人从"重复造轮子 + 踩重复的坑"中解放出来,显著缩短大模型上线时间。**而且因为只下发一条
命令、由脚本内部按 rank 自分角色,这套方式在只有单一部署单元的推理 1.0 上同样适用**——
正好补上了 1.0 现在无法多角色部署的短板。

## 四、我们已经有一套可直接复用的参考实现

上面两点不是设想,我们已经把它落地并验证:

- **`setup_rank_env.sh`** —— rank table 监听 + 解析 + 注入上述环境变量(即"建议一"的实现);
- **`run.sh`** —— 统一入口,source 上面的脚本后,按 rank `exec` 到对应角色的启动脚本
  (即"建议二"的骨架);
- 在 16 套不同模型 / 平台 / 拓扑的部署上复用同一套骨架,验证了它的通用性。

这套实现可以直接作为产品规格的参考。(如需,我们可以提供完整代码与文档。)

## 五、为什么值得排进产品规划

- **减少部署失败与支持工单**:握手超时、KV 端口冲突、group 0 路由配错这类问题,绝大多数
  源于用户各自手写编排,平台统一承担后可成批消除。
- **加快大模型推理上线**:用户只填模型参数即可部署,onboarding 时间显著缩短。
- **消除一整类用户侧 bug**:包括各自手写解析的隐蔽错误、脚本在多节点间的不一致漂移等。
